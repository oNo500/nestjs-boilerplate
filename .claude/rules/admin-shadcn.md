# Frontend Rules (admin-shadcn)

Architectural rules for the Next.js admin panel: feature organization, UI library usage, API typing, and style conventions.

## Tech Stack

- **Framework**: Next.js 16 (App Router · Turbopack)
- **Language**: TypeScript
- **UI**: `@workspace/ui` (built on `@base-ui/react`) · shadcn · Tailwind CSS v4
- **Data Fetching**: TanStack Query v5 · openapi-fetch · openapi-react-query
- **Forms**: React Hook Form · Zod · @hookform/resolvers
- **Tables**: @tanstack/react-table
- **Animation**: Motion
- **Testing**: Vitest · @testing-library/react · MSW v2
- **Env**: @t3-oss/env-nextjs
- **Toolchain**: pnpm · ESLint · Turbopack

## Architecture: Feature-Based

```
src/
├── app/              # Next.js App Router (routing only, no business logic)
├── features/         # Business modules, one per feature
│   └── {feature}/
│       ├── components/   # Feature-local components (optional)
│       ├── hooks/        # Feature-local hooks (optional)
│       └── {entry}.tsx   # Feature entry component
├── components/       # Cross-feature shared components
├── hooks/            # Cross-feature shared hooks
├── lib/              # Third-party wrappers (api-client, query-client, etc.)
├── config/
│   ├── env.ts        # Env variables (via @t3-oss/env-nextjs)
│   └── app-paths.ts  # Centralized route paths
└── testing/          # Test utilities (renderWithProviders, fixtures)
```

## Feature Boundaries

### Partition rules

A feature typically maps 1:1 to a backend context.

**Fold into existing feature** — all conditions must hold:

1. **Same backend context**: consumes the same group of backend APIs
2. **Shared data model**: primary data model and view state stay within this feature
3. **No standalone entry**: no new top-level route or navigation item required

**Create new feature** — any one condition triggers a new feature:

1. **Independent backend context**: API set is largely disjoint from existing features
2. **Top-level route**: owns a first-level URL segment (e.g. `/users` vs `/roles`)
3. **Independently disposable**: can be removed without affecting other features

### Shared component promotion

When ≥ 2 features need the same component, promote it to `src/components/` rather than creating a new feature for it.

### Feature naming

- Directories use `kebab-case` (see the global `Conventions` table)
- Collections use plural (`articles`, `audit-logs`); workflows use singular (`auth`, `settings`)

## UI Library (`@workspace/ui`)

Components are built on `@base-ui/react` and **do not** support Radix-style `asChild`:

- `DropdownMenuTrigger` uses the `render` prop
- `Button` is composed via style anchor
- Components in `packages/ui` are managed via `shadcn cli` and not edited by hand

## API Types

- Generated by `openapi-typescript` into `packages/api-types/src/openapi.d.ts`
- Consumed via the `@workspace/api-types` package; never hand-written
- Regenerate after API changes: `pnpm --filter @workspace/api-types api:gen`

## Conventions

| Scope | Rule | Example |
|---|---|---|
| File / directory | kebab-case | `user-profile.tsx` |
| Component | PascalCase | `UserProfile` |
| Function / variable | camelCase | `getUserData` |
| Type / interface | PascalCase | `User`, `AuthConfig` |
| Constant | UPPER_SNAKE_CASE | `API_BASE_URL` |

## Prohibited

- Business logic or complex JSX in `app/` route files (push down into `features/`)
- `index.ts` barrel files under `features/` (import from source directly)
- Hard-coded route paths in components (use `config/app-paths.ts`)
- Hand-written API types (import from `@workspace/api-types`)
- Hard-coded role strings (import `RoleType` / `ROLES` / `hasRequiredRole` from `@/lib/rbac`; use `<RequireRole>` / `<ShowForRole>` for guards)
- Manual edits inside `packages/ui`
- Pre-creating empty subdirectories
